# File Sharing App

This project is a simple file sharing application built with Node.js, Express, MongoDB, Multer, EJS, and Nodemailer. It lets a user upload a file, get a shareable download link, and optionally send that link to another person by email.

The app is split into two parts:

- `Backend/` - the server, database connection, API routes, file upload logic, email sending logic, and the download page template.
- `FrontEnd/` - the public upload UI that lets a user drag and drop a file, copy the generated link, and send it through email.

## What The App Does

The application follows this flow:

1. A user opens the frontend page.
2. The user selects or drops a file.
3. The browser uploads the file to the backend using `multipart/form-data`.
4. The backend stores the file on disk and saves file metadata in MongoDB.
5. The backend returns a unique share link.
6. The user can copy the link or send it by email.
7. The recipient opens the link and sees a download page.
8. The recipient can download the file directly.

## Main Features

- Drag-and-drop file upload.
- Manual file selection through the file picker.
- Upload progress indicator in the browser.
- Unique shareable link for each uploaded file.
- Download page rendered with EJS.
- Optional email sharing with a generated download link.
- File metadata stored in MongoDB.
- Files saved on the server filesystem inside the `uploads/` folder.

## Tech Stack

- Node.js - runtime
- Express - HTTP server and routing
- MongoDB + Mongoose - file metadata storage
- Multer - file upload handling
- EJS - server-side rendering for the download page
- Nodemailer - sending email with the file link
- UUID - generating unique share IDs
- Vanilla HTML, CSS, and JavaScript - frontend UI

## Backend Overview

### `server.js`

The main server entry point does the following:

- creates the Express app,
- serves static files from `Backend/public`,
- connects to MongoDB through `config/db.js`,
- enables JSON and URL-encoded request parsing,
- configures EJS as the view engine,
- mounts the file API routes at `/api/files`,
- mounts the download page routes at `/files`,
- starts listening on `PORT` or `3000`.

### Database Connection: `config/db.js`

This file loads environment variables using `dotenv` and connects to MongoDB with `mongoose.connect(process.env.MONGO_CONNECTION_URL)`.

If the connection fails, the process exits with code `1`.

### File Model: `models/file.js`

The MongoDB schema stores:

- `filename` - the saved file name
- `path` - the local path on disk
- `size` - file size in bytes
- `uuid` - unique public identifier used in share links
- `sender` - email address of the person who sent the file link
- `receiver` - email address of the recipient

The schema uses timestamps, so `createdAt` and `updatedAt` are added automatically.

## API Routes

### `POST /api/files`

This route uploads a file.

Behavior:

- accepts one file field named `myfile`,
- stores the file in `Backend/uploads/`,
- creates a MongoDB record with the file metadata,
- generates a UUID for the share link,
- returns a JSON response containing the public link.

Example response:

```json
{
	"file": "http://your-app-url/files/uuid-here"
}
```

Important details from the code:

- the upload limit is set to `1000000 * 100`, which is approximately 100 MB,
- the file is saved with a unique timestamp-based filename,
- the response uses `process.env.APP_BASE_URL` to build the public URL.

### `POST /api/files/send`

This route sends the file link by email.

Expected body:

```json
{
	"uuid": "file-uuid",
	"emailTo": "receiver@example.com",
	"emailFrom": "sender@example.com"
}
```

Behavior:

- validates that all fields are present,
- finds the file record by UUID,
- prevents sending again if the sender is already stored,
- saves `sender` and `receiver` values to MongoDB,
- sends an email containing the download link.

The email uses:

- subject: `File shared using app`
- text: a short plain-text message
- html: a custom template from `Routes/services/emailTemplate.js`

### `GET /files/:uuid`

This route renders the download page.

Behavior:

- looks up the file by UUID,
- renders `views/download.ejs`,
- shows file name, file size, and the download button,
- displays an error message if the link is missing or expired.

### `GET /files/download/:uuid`

This route downloads the actual file.

Behavior:

- looks up the file record by UUID,
- returns `res.download(file.path)`,
- sends a 404 if the file metadata cannot be found.

## Email Service

### `Routes/services/emailService.js`

This helper creates a Nodemailer transporter and sends the file-sharing email.

It expects these environment variables:

- `MAILTRAP_HOST`
- `MAILTRAP_PORT`
- `MAIL_USER`
- `MAIL_PASS`

The code currently uses a Mailtrap-style setup, even though the note in the old README mentions Brevo. If you want to use a different provider, this is the file to update.

### `Routes/services/emailTemplate.js`

This file returns the HTML body for the email. It shows:

- who sent the file,
- the approximate file size,
- the link expiration window,
- a download button.

## Frontend Overview

### `FrontEnd/index.html`

The frontend page contains:

- a drop zone for drag-and-drop uploads,
- a hidden file input for browsing files,
- a progress UI while the upload is running,
- a share-link field with a copy-to-clipboard icon,
- an email form for sending the link.

### `FrontEnd/script.js`

This script handles the entire client-side flow:

- drag-and-drop upload handling,
- browsing and selecting files,
- XMLHttpRequest upload with progress tracking,
- copying the generated share link,
- submitting the email form to `/api/files/send`,
- showing toast messages for success and errors.

Important client-side constants:

- `host` is set to `https://inshare.herokuapp.com/`
- `uploadURL` points to `api/files`
- `emailURL` points to `api/files/send`
- maximum allowed file size is set to `100 MB`

### `FrontEnd/style.css`

This file styles the upload card, drag-and-drop area, progress indicator, share panel, email form, and responsive mobile behavior.

## Download Page

### `Backend/views/download.ejs`

This EJS template is the public page for anyone opening a share link.

It shows:

- the app logo,
- the file name,
- the file size,
- a message that the link expires in 24 hours,
- a direct download button,
- an error state when the file is unavailable.

## Environment Variables

Create a `.env` file in the `Backend/` folder with values similar to the following:

```env
PORT=3000
APP_BASE_URL=http://localhost:3000
MONGO_CONNECTION_URL=mongodb://127.0.0.1:27017/file_sharing_app
MAILTRAP_HOST=sandbox.smtp.mailtrap.io
MAILTRAP_PORT=2525
MAIL_USER=your_email_user
MAIL_PASS=your_email_password
```

Notes:

- `APP_BASE_URL` is used to build public links in API responses, emails, and download pages.
- `MONGO_CONNECTION_URL` must point to a running MongoDB instance.
- The mail settings must match the provider you are using.

## Installation

1. Install Node.js and MongoDB.
2. Open a terminal in `Backend/`.
3. Install dependencies:

```bash
npm install
```

4. Create the `.env` file with the variables above.
5. Make sure the `uploads/` folder exists and is writable.

## Running The App

From the `Backend/` folder:

```bash
npm run dev
```

or

```bash
npm run server
```

The server will listen on the configured port, or on `3000` by default.

## How To Use It

### Upload a file

1. Open the frontend page.
2. Drop a file into the drop zone or click Browse.
3. Wait for the upload to finish.
4. Copy the generated link or send it by email.

### Download a file

1. Open the shared link.
2. Review the download page.
3. Click Download file to retrieve the uploaded file.

### Send by email

1. Upload a file first.
2. Fill in your email and the recipient's email.
3. Submit the form.
4. The recipient receives a message with the download link.

## Data Flow Summary

The app moves data through three layers:

- the browser handles file selection, progress, link copying, and email submission,
- the backend accepts uploads and renders the download page,
- MongoDB stores the file metadata so the app can resolve UUIDs back to files on disk.

## Known Caveats And Notes

- The frontend host is hardcoded to `https://inshare.herokuapp.com/`, so local development may require changing that value.
- The old note mentioned Brevo, but the actual code is configured like a Mailtrap setup.
- The file-size check in the frontend compares `selectedFiles[0].length`, while file objects normally use `size`; this should be reviewed if uploads larger than expected are not being blocked correctly.
- The download link is described as expiring in 24 hours in the UI and email template, but there is no automatic expiration cleanup in the backend code shown here.
- The app stores uploaded files on the local filesystem, so deployment environments must keep the `uploads/` directory persistent if you want downloads to keep working.

## Scripts

From `Backend/package.json`:

- `npm run dev` - starts the server with nodemon
- `npm run server` - starts the server with Node directly

## Suggested Improvements

If you want to extend the app later, the most useful additions would be:

- automatic cleanup for expired files,
- a configurable app base URL for development and production,
- file type restrictions,
- better upload validation on the frontend and backend,
- persistent storage for production deployments,
- clearer error messages for email delivery failures.

## Summary

This project is a lightweight file sharing tool. A user uploads one file, gets a unique link, and can either share that link directly or send it by email. The backend stores metadata in MongoDB, the file itself on disk, and the frontend provides the upload and sharing experience.
